The real cost of a 1MB JavaScript bundle

9 min read · January 2025

A 1MB JavaScript bundle doesn't just cost you 1MB of transfer. It costs you parse time, compile time, and execution time — and those costs scale with device capability, not connection speed. That's why the same bundle can feel fast on a MacBook and unusable on a mid-range Android.

The three costs people forget

Transfer size is what most developers look at. But after the bytes arrive, the browser has to parse the text into an AST, compile it to bytecode, and then execute it. On a Moto G4, that process for a 1MB bundle can take 4–5 seconds. Gzipping gets you smaller transfer, but it doesn't reduce parse and compile time — those operate on the uncompressed size. 

// Measure script execution time
const start = performance.now();
// ... your code
const end = performance.now();
console.log(`Execution: ${end - start}ms`);

Code splitting is not optional

If you're shipping a monolithic bundle, you're paying the full parse and compile cost on every page load, even for code that page will never run. Route-based code splitting is table stakes. Dynamic imports for heavy features (charts, editors, date pickers) are the next step.

What actually moves the needle

In order: remove unused dependencies first (bundle-analyzer will shock you), then split routes, then lazy-load heavy features. Tree shaking helps but it's often overstated — it can't remove code that's genuinely imported, only code that's never referenced. Most real-world bundles have more of the former.
