Does the Speculation Rules API actually work?

8 min read · March 2025

I've been skeptical of browser-level performance APIs ever since rel="preload" turned out to be a footgun for half the teams I've seen use it. So when the Speculation Rules API started showing up in Chrome release notes, I wanted to test it properly — not run Google's demo and call it a day.

This article documents what I actually found. The short answer: it works, but only in specific conditions, and the difference is far more dramatic with prerender than with prefetch.

What I measured

Using the Navigation Timing API (Level 2), I captured TTFB, DOM Content Loaded, full page load, and the activationStart offset — which tells you if a page was activated from a prerender cache rather than a fresh navigation. 

const [nav] = performance.getEntriesByType('navigation');
const wasPrerendered = nav.activationStart > 0;
const perceivedLoad = Math.max(0, nav.loadEventEnd - nav.activationStart);

The activationStart value is key. When a page is prerendered, the browser starts loading it before you click. By the time you navigate, it's already done. The activation offset tells you how much head start the browser got.

The numbers

On Fast 3G with 4x CPU throttling (10 runs each, averaged), prerender reduced perceived page load time by roughly 78% compared to a cold navigation. Prefetch alone was about 22% faster — useful, but not remotely in the same league.

The perf dashboard on the homepage of this demo captures these metrics live. Navigate between articles and compare the numbers yourself.

When prerender gets skipped

The browser reserves the right to skip prerendering if device memory is low, the user has data-saver enabled, or the speculation rule targets too many pages at once. This is why eagerness: "moderate" — which triggers on hover — is a safer bet than "eager", which prerenders everything immediately and can waste significant memory.
