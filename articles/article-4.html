Caching

HTTP caching headers: what every developer gets wrong

10 min read · December 2024

HTTP caching is one of those things where the basic concept is simple, but most teams get the details wrong in production and end up either caching too aggressively (users see stale content) or not at all (they pay full round-trip cost on every request). Both are fixable.

Cache-Control isn't one directive

Most developers know max-age. Far fewer understand stale-while-revalidate, which is probably the most useful directive for dynamic content. It lets the browser serve a cached response immediately while fetching a fresh one in the background — so the user never waits, and they get updated content on the next load. 

Cache-Control: max-age=60, stale-while-revalidate=300

This tells the browser: serve from cache for 60 seconds. After that, it's stale — but you can still serve it for up to 5 minutes while refreshing in the background. After 5 minutes, go to the network.

ETags vs. Last-Modified

Both are conditional request mechanisms — they let the browser ask the server "has this changed?" instead of fetching the full response. ETags are more precise (they hash content rather than relying on timestamps), but they require server-side computation. For static assets, both work fine. For dynamic content, ETags are the safer choice.

The immutable trick for static assets

If you fingerprint your assets (which every modern bundler does by default), you can set Cache-Control: max-age=31536000, immutable — one year, and tell the browser it will never change. The browser won't even bother with a conditional request. This is the right setup for versioned JS/CSS files.
